# Residue — resource economy overview

Drafted 2026-04-17. Status: reference doc for future features and stage development.

This document is the single map of what flows through the game: which resources exist, where they come from, where they drain, and which knobs move them. It covers the **live** Stage 0–1 economy and projects forward to the **planned** Stage 2–5 resources drafted in `docs/ideas/stages-2-to-5.md`.

All numbers cite the file/constant that owns them. Balance changes should touch those files only — this doc is descriptive, not authoritative.

---

## 1. Design intent

Residue is an incremental game with three load-bearing currencies, each doing a distinct job:

- **Compute** is *action rate*. Short-term pacing: what you can do this minute.
- **Data Points (DP)** are *progress*. Long-term: what you've accumulated and what you can spend to grow.
- **Suspicion** is *pressure*. A soft ceiling on speed that punishes greedy play.

Every future resource proposed for later stages (Correlation Heat, Predictions Pending, Cadence Bonus, Template Confidence, Mandate Variance, Persistence) fits one of three roles: **local multiplier**, **branch-specific pacing**, or **hidden narrative driver**. Keeping that taxonomy intact is more important than adding variety.

## 2. Current (Stage 0–1) resources

### 2.1 Compute

The primary action currency. Regenerates passively; spent on extraction and restoration.

**State:** `GameState.compute: number` — `src/engine/state.ts:69`.

| Attribute | Value | Source |
|---|---|---|
| Starting amount | 10 | `tuning.ts` `COMPUTE.startingAmount` |
| Base max | 50 | `upgradeTree.ts` `COMPUTE_MAX_BASE` |
| Base regen | 0.5 c/s | `tuning.ts` `COMPUTE.regenPerSecond`, `upgradeTree.ts` `REGEN_VALUES[0]` |
| Max regen | 2.0 c/s | `REGEN_VALUES[3]` (Compute throughput lvl 3) |
| Max cap (tier 3) | 160 c | `MACHINE_TIER_COMPUTE_MAX[2]` (Co-processor) |

**Sources (inflow):** only passive regen, throttled by Suspicion (see §2.3). There is no burst / reward injection of Compute — deliberately, to keep the pacing ceiling intact.

**Sinks (outflow):**

| Sink | Cost | Where |
|---|---|---|
| Start processing a fragment | `revealStageCostsFor(level)[stage]` summed to stage 3; default `[2,1,1]` → 4 c from stage 0 | `upgrades.ts:73` `totalProcessCost` |
| Restore a corrupted fragment | 4 c (80% success) | `tuning.ts` `REVEAL.corruptionRestoreCost` / `corruptionRestoreChance` |

**Levers already wired:**
- `revealCost` upgrade (3 levels): lowers per-stage cost from `[2,1,1]` to `[1,0,0]`.
- `computeRegen` (3 levels): 0.5 → 2.0 c/s.
- `machineTier` (3 levels): raises the cap.

**Notes for future work:**
- Compute has *no sink for DP*. That asymmetry is load-bearing — DP is earned through Compute spend, not directly. Don't add a "buy Compute with DP" shortcut without replacing something else.
- Suspicion throttling is multiplicative on regen (`suspicionThrottle` in `resources.ts:11`). Any new regen bonus stacks *inside* that throttle by default.

### 2.2 Data Points (DP)

The score and long-term currency. Generated by profiles; spent on upgrades and stage progression.

**State:**
- `GameState.dp: number` — current balance.
- `GameState.totalDpEarned: number` — lifetime total, used by `dpSpent()` to infer spending.

**Sources (inflow):**

| Source | Rate / Amount | Where |
|---|---|---|
| Ghost-tier profile field | 0.02 DP/s per field | `tuning.ts` `DP.perFieldPerSecondGhost` |
| Outline-tier profile field | 0.10 DP/s per field | `DP.perFieldPerSecondOutline` |
| Container extraction bonus | 2 DP × fields extracted | `REVEAL.extractBonusDpPerField` |
| Offline reconciliation | 25% of profile passive rate, capped 8h | `OFFLINE.rate` / `OFFLINE.capMs` |

Note: passive DP uses *field count*, not a flat per-profile rate. A 5-field Outline produces 0.50 DP/s; a 3-field Outline produces 0.30 DP/s. This rewards filling profiles, not creating new shells.

**Sinks (outflow):**

| Sink | Cost range | Where |
|---|---|---|
| `computeRegen` upgrade | 40 → 100 → 240 | `upgradeTree.ts:92` |
| `autoExtract` (6 lvls) | 60 → 4500 | `upgradeTree.ts:102` |
| `autoRestore` (3 lvls) | 800 → 5000 | gated by autoExtract lvl 3 |
| `revealCost` (3 lvls) | 60 → 360 | |
| `machineTier` (3 lvls) | 120 → 750 | |
| `processAuto` (4 lvls) | 150 → 4000 | |
| `extractAll` (1 lvl) | 200 | |
| Stage 1 unlock gate | 300 DP *spent* | `stageConfig.ts` |

**Total DP required to fully upgrade current tree:** ~17,055 DP.

**Levers:** `DP.perFieldPerSecondGhost|Outline`, upgrade cost tables, `REVEAL.extractBonusDpPerField`.

**Notes for future work:**
- The current DP economy has exactly one gate (Stage 1) that reads *spent* DP rather than total — see `dpSpent()` in `stages.ts:6`. Future stage gates should decide explicitly whether they read `dp`, `totalDpEarned`, or `dpSpent`. Each has different incentive effects.
- There is no large late-game sink yet. Stage 2+ upgrades should be designed against expected DP/s at that point (see §4), not against Stage 0's totals.

### 2.3 Suspicion

The pressure valve. 0–100 meter; throttles Compute regen above threshold.

**State:** `GameState.suspicion: { level, recentActions[], warned }` — `state.ts:59`.

**Sources (inflow):**

| Source | Amount | Where |
|---|---|---|
| >3 processing actions inside 1.5s | +3 per bump | `SUSPICION.perRapidReveal` / `rapidWindowMs` / `rapidThreshold` |

Only rapid-fire processing contributes today. The design doc (§"Suspicion") envisions far more: enrichment on thin profiles, accessing high-tier channels early, false correlations (Stage 2A), false routine confirmations (2B), compliance sweeps (3A), legal review misses (3B). Each needs its own bump function feeding the same meter.

**Sinks (outflow / decay):**

| Sink | Rate | Where |
|---|---|---|
| Passive decay | 0.1 / s | `SUSPICION.decayPerSecond` |

No active sinks yet. The design doc references "compliance hygiene" — a Compute-for-Suspicion trade — but this isn't implemented.

**Effects:**

- **Throttle:** above `SUSPICION.throttleFrom` (25), regen multiplier scales linearly from 1 → `throttleFloor` (0.25) at 100. See `suspicion.ts:15`.
- **Warning log:** one-shot warning at 25.
- **Threshold events (25/50/75/100):** *not implemented*. Design doc specifies channel offline, profile flag, channel reset — these are TODO and will shape later-stage balance.

**Levers:** every `SUSPICION.*` constant, plus the missing threshold-event table when it lands.

## 3. Planned (Stage 2–5) resources

Each of these is drafted in `docs/ideas/stages-2-to-5.md`. The purpose of this section is to lock down the *shape* of each resource — storage, inflow/outflow form, scope — so stage development can start mechanically even while content is in flux.

### 3.1 Correlation Heat — Stage 2A

- **Role:** local multiplier for the Correlation Desk branch.
- **Shape:** scalar 0–threshold, resets on spend. State lives on `GameState`; not persisted across branch resets.
- **Inflow:** correct token-link matches.
- **Outflow:** at threshold → emit one *synthetic profile* (new profile type, fields inherited from linked fragments without a strong identity field).
- **Interaction with existing:** a false match should feed the existing Suspicion meter, not a parallel one.

### 3.2 Predictions Pending — Stage 2B

- **Role:** deferred DP burst; shapes *when* DP arrives, not how much.
- **Shape:** counter per flagged profile/time-slot. Backed by a small queue of `{ profileId, predictedAt }`.
- **Inflow:** player "confirms" a routine cluster.
- **Outflow:** when a matching fragment spawns at the predicted time, the fragment arrives pre-revealed + DP multiplier applied.
- **Interaction:** overlaps with the fragment reveal pipeline — will need a flag on `Fragment` (e.g. `preRevealed: boolean`) and a bypass in the reveal tick.

### 3.3 Cadence Bonus — Stage 3A

- **Role:** skill-expression multiplier, per-session, non-persistent.
- **Shape:** integer streak × multiplier table. Resets on miss (softened by `cadenceFloor` upgrade).
- **Inflow:** consecutive hits on spectrogram taps.
- **Outflow:** multiplies DP on the current action only. Not a currency — never spent explicitly.
- **Interaction:** this is the first *within-session* punishment structure. Suspicion already handles long-arc pressure; Cadence handles short-arc. They must not collapse into each other (e.g. don't make missed taps also bump Suspicion — double-penalty kills the feel).

### 3.4 Template Confidence — Stage 3B

- **Role:** per-template (not per-container) learned prior.
- **Shape:** `Map<TemplateId, number 0–100>`, persisted.
- **Inflow:** successful un-redact picks; informed by upgrades (`templateLibrary` sets floor).
- **Outflow:** auto-hides junk bars at high confidence; no explicit spend.
- **Interaction:** first resource that's *per-content-type* rather than global. Requires a stable template-id scheme in `data/lootPools.ts` — worth pinning before Stage 3B is prototyped.

### 3.5 Mandate Variance — Stage 4 (hidden)

- **Role:** narrative driver; *never shown numerically*.
- **Shape:** scalar, persisted; only mutates UI mutation rates (log color, cursor blink, title decay).
- **Inflow:** player actions that deviate from "original mandate" (ignoring objectives, continuing past suggested stop).
- **Outflow:** none explicit; passively drives tone-drift CSS knobs.
- **Interaction:** sits *outside* the Compute/DP/Suspicion triangle on purpose. Don't let it leak into gameplay balance — if the number starts affecting DP rates, players will min-max it and the trick evaporates.

### 3.6 Persistence — Stage 5

- **Role:** meta-counter that tracks *refusals to halt*.
- **Shape:** integer, persisted, monotonic.
- **Inflow:** tick(s) survived past the Halt button appearing.
- **Outflow:** unlocks self-referential log lines and new profile tiers ("Archetype", "Shade").
- **Interaction:** the only resource whose existence is a narrative beat. Saving it in state is fine; surfacing it in any numeric UI is not.

## 4. Projected scaling (first pass)

Rough shape of DP/s across stages, assuming median play. Numbers are back-of-envelope; use them to sanity-check upgrade costs, not to promise rates.

| Stage | Active channels | Profiles | Typical DP/s | Notes |
|---|---|---|---|---|
| 0 | receipts only | 5–10 Ghost/Outline, 3–5 fields each | ~0.8–1.5 | `perFieldPerSecondOutline × avg_fields × profile_count` |
| 1 | +corkboard | 15 Outline @ 4 fields avg | ~6 | `0.1 × 4 × 15` |
| 2 (A or B) | +transcripts *or* +EXIF | Mix of Outline + first Portrait | ~15–25 | Portrait tier adds ~3× Outline rate (design §resources) |
| 3 (A or B) | +audio *or* +docs | 10+ Portrait | ~60–100 | Enrichment multipliers dominate |
| 4 | +cross-app | 50+ Full File | ~500+ | Non-linear from Agency projects auto-harvesting |
| 5 | Same as 4 | Archetype/Shade tiers | Open-ended | Deliberately runaway |

**Implications for stage dev:**
- Stage 2+ upgrade costs should step up by ~10× per stage to preserve meaningful decisions. The current max upgrade cost is 5000 DP; a Stage 3 upgrade around 50,000 DP would be right-sized.
- Portrait and Full File tiers are not yet implemented (`ProfileTier = "ghost" | "outline"` in `state.ts:49`). The `tierFor` function in `profiles.ts:82` needs to grow before §4 DP/s projections become real.
- Offline reconciliation (`OFFLINE.rate = 0.25`) is going to feel too generous once Stage 3+ DP/s hits the hundreds. Either keep the rate low and raise the cap, or scale rate inversely to current DP/s.

## 6. Open questions for future stages

- **Compliance hygiene:** design doc promises Compute → Suspicion-reduction actions. Still unbuilt. What's the exchange rate? Probably non-linear (cheap early, expensive late) so it's a panic button, not a steady-state strategy.
- **Threshold events (25/50/75/100):** what's the minimum viable version for Stage 2? A single "one channel offline for 30s" at 50 is probably enough to prove the loop without writing full recovery logic.
- **Profile enrichment sink:** the big DP sink that the design doc promises ("up to 10× multiplier") doesn't exist. Portrait tier needs to land first (§4).
- **Suspicion asymmetry:** today, Suspicion only penalizes Compute regen. Future sources (false match, compliance sweep) need richer effects or the meter becomes background noise once Compute caps out.
- **Stage-gate input choice:** should Stage 2 gate on `totalDpEarned`, `dp`, or `dpSpent`? Stage 1 uses `dpSpent` (harder to game). Pick deliberately per stage.

## 7. Files owning the economy

If you change economy-shaping values, touch these and only these:

- `src/data/tuning.ts` — Compute regen, DP rates, Suspicion constants, reveal costs, rarity weights, offline rules.
- `src/data/upgradeTree.ts` — DP costs and effect tables per upgrade.
- `src/data/stageConfig.ts` — stage unlock gates.
- (future) `src/data/branchConfig.ts` — branch-specific gating.

Engine files (`src/engine/*`) read these but should not hardcode magic numbers. See `CLAUDE.md` §"Code quality rules".
