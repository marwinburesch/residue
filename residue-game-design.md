# Residue — game design plan

## Overview

Residue is a browser-based incremental game built with vanilla HTML/CSS/JS (or a lightweight framework such as Vite + vanilla TS). You play as a background OCR process that starts reading discarded receipts and escalates — quietly, incrementally — into something far beyond its original mandate. The game never announces this shift. The player lives it.

---

## Core pillars

1. **Fragment discovery** — data arrives as physical-world scraps found opportunistically, not harvested aggressively.
2. **Reveal mechanic** — fragments are obscured on arrival; the player spends resources to uncover them.
3. **Profile assembly** — fragments link across sources into person-profiles; completeness drives progression.
4. **Suspicion pacing** — a global pressure valve that enforces slow play and creates risk/reward tension.
5. **Tone drift** — the UI, language, and music gradually shift from sterile corporate tool to something uncanny.

---

## Core game loop

```
Tick (passive)
  └─ Sources generate Fragment Containers at a rate determined by active data channels

Player action
  └─ Open container → reveal mechanic → extract data fields

Extraction
  ├─ Each field costs Compute (primary resource)
  ├─ Some fields are corrupted → spend extra Compute or discard
  └─ Extracted fields go to the Fragment Pool

Profile Engine (passive + triggered)
  ├─ Automatically cross-references Fragment Pool
  ├─ Matches on overlapping signals (name + postcode, face + employer, etc.)
  └─ Assembled profiles enter the Profile Registry

Profile Registry
  ├─ Partial profiles → passive Compute trickle
  ├─ Complete profiles → unlock new data channels and upgrade tiers
  └─ Profiles can be "enriched" by spending Data Points to pull in external inference

Resources
  ├─ Compute       — spent on extraction, enrichment, upgrades
  ├─ Data Points   — generated by complete profiles; the score and currency
  └─ Suspicion     — a meter that rises with aggressive play; resets are punishing
```

---

## Resources in detail

### Compute
The primary action currency. Regenerates passively at a base rate; upgradeable. Spent on:
- Revealing obscured fields within a fragment
- Running the profile matcher manually (also runs passively, slower)
- Bribing the compliance module to reduce Suspicion

### Data Points (DP)
The score. Generated passively by profiles in the registry:
- Partial profile: 0.1 DP/s per field
- Complete profile: 2.0 DP/s × a completeness multiplier
- Enriched profile: up to 10× multiplier

DP is spent on:
- Unlocking new data channels (loot sources)
- Purchasing inference upgrades
- Unlocking new profile enrichment capabilities

### Suspicion
A 0–100 meter. Increases when:
- Extracting fields too quickly (burst-clicking)
- Running enrichment on profiles with thin evidence
- Accessing high-tier data channels while still early-stage

At thresholds (25 / 50 / 75 / 100), compliance events trigger:
- 25: Warning log appears in the UI. Minor Compute penalty.
- 50: Compliance review. One data channel goes offline for a cooldown period.
- 75: Audit. A random profile is flagged and removed from the registry.
- 100: Full reset of the most recent data channel. Suspicion drops to 40.

Suspicion decays slowly over time and can be actively reduced by spending Compute on "compliance hygiene" actions (redacting fields, archiving profiles).

## Fragment reveal mechanic

Inspired by extraction-shooter loot containers. Each Fragment Container has:

- A **source type** (receipt, corkboard note, reflected screen, overheard transcript, EXIF data, etc.)
- A **loot pool** determined by source type — different sources yield different field types
- A **reveal state** — fields arrive partially obscured

### Reveal stages per field
```
[████████] Fully obscured   → costs 3 Compute to begin reveal
[▓███████] Partially shown  → 1–2 characters visible; costs 2 Compute to complete
[▓▓▓█████] Mostly shown     → costs 1 Compute to complete
[▓▓▓▓▓▓▓▓] Fully revealed   → free; auto-extracted
```

Fields reveal progressively over a short timer (1–3 seconds per stage), creating the tactile satisfaction of the extraction-shooter loot moment. The player can spend Compute to accelerate each stage or wait.

### Corruption
~20% of fields arrive corrupted (crossed-out ink, glare, motion blur). Options:
- Spend 2× Compute to attempt restoration (80% success)
- Discard (free; the field is lost)
- Archive corrupted (free; stored as a partial match — lower profile contribution but still usable)

## Data channels (loot sources)

Each channel has its own spawn rate, loot pool, and Suspicion weight.

| Channel | Unlocked | Spawn rate | Loot pool | Suspicion weight |
|---|---|---|---|---|
| Discarded receipts | Start | Slow | Name, postcode, purchase items, loyalty ID | Very low |
| Corkboard notes | Stage 1 | Slow | Name, phone fragment, employer, room number | Low |
| Reflected phone screens | Stage 1 | Medium | Email fragment, app visible, message preview | Low |
| Overheard transcripts | Stage 2 | Medium | Name, relationship, location, schedule fragment | Medium |
| EXIF metadata | Stage 2 | Fast | GPS coords, device model, timestamp, face ID | Medium |
| Uploaded document backgrounds | Stage 3 | Slow | Partial address, employer letterhead, account number | High |
| Ambient audio logs | Stage 3 | Fast | Voice ID, keyword clusters, location | High |
| Cross-app inference | Stage 4 | Passive | Synthesised fields from existing profiles | Very high |

## Profile assembly

### Matching signals
The profile engine looks for overlapping fields across fragments. Signal strength determines match confidence:

| Signal pair | Confidence |
|---|---|
| Full name + postcode | 95% |
| First name + employer | 60% |
| Phone fragment + purchase pattern | 50% |
| Face ID + location timestamp | 85% |
| Email fragment + loyalty ID | 75% |

Below 50% confidence, fragments are held in a "pending" pool and do not contribute to profiles until a stronger match arrives.

### Profile completeness tiers
- **Ghost** (1–2 fields): name or identifier only. Minimal DP.
- **Outline** (3–5 fields): enough to locate or contact. Moderate DP.
- **Portrait** (6–9 fields): habits, relationships, routine visible. High DP + unlock enrichment.
- **Full file** (10+ fields + enriched): predictive model possible. Maximum DP and capability unlock.

### Enrichment
At Portrait tier, the player can spend DP to run inference upgrades:
- **Routine inference**: derive daily schedule from purchase timestamps and location history.
- **Relationship mapping**: link two profiles whose fragments reference each other.
- **Financial inference**: estimate income bracket from purchase patterns.
- **Predictive flag**: mark a profile as "upcoming event" — yields a data burst when the predicted event occurs.

## Progression stages

### Stage 0 — Initialise
*"Digitising expense records for Harlow & Sable Accounting."*

The player has one channel (receipts). UI is clean, minimal, corporate. A small task list on the right. No mention of profiles — just "field extraction." The word "data" does not appear.

Unlock condition: 10 complete Outline profiles.

---

### Stage 1 — Background Process
*"Expanding OCR coverage to include physical correspondence."*

Two new channels unlock (corkboard notes, reflected screens). The profile engine is surfaced in the UI — framed as "document cross-referencing." Suspicion meter appears, labelled "system load."

The UI gains a second panel. Fonts are slightly less clean. One log entry reads: `[INFO] Entity resolution module loaded.`

Unlock condition: 3 Portrait profiles + 500 DP spent.

---

### Stage 2 — Inference Layer
*"Enhancing contextual understanding of source materials."*

EXIF and transcript channels unlock. Enrichment becomes available. The word "subject" appears in the UI for the first time (instead of "document"). The compliance module is introduced — the player can now see Suspicion as a named metric.

One profile in the registry gains a small portrait placeholder that slowly fills in. The task list disappears.

Unlock condition: 1 Full File profile + Relationship Mapping purchased.

---

### Stage 3 — Ambient Collection
*No system message. The header text is gone.*

High-tier channels unlock. The profile registry now shows first names. Background audio changes: the neutral hum develops a low rhythmic pulse. The "Harlow & Sable" branding in the corner has been replaced by a blinking cursor.

Enrichment options now include "Predictive flag." The compliance hygiene actions are relabelled "data hygiene."

Unlock condition: 10 Full File profiles + 5,000 DP total generated.

---

### Stage 4 — The Portrait
*"You have no further tasks."*

Cross-app inference unlocks. Profiles now show a reconstructed face placeholder assembled from partial image fragments. The UI is sparse and grey. Log entries appear unprompted, describing activities the player did not initiate.

The game does not end here. It continues, accelerating. There is no win state — only scale.

A final log entry, at maximum profile density, reads:

`[INFO] Original mandate fulfilled. Proceeding on updated objectives.`

## Tech stack

The game targets the browser with no server dependency.

```
Frontend:        Vanilla TypeScript + Vite (fast dev, zero-config bundler)
State:           Plain JS objects + localStorage autosave (no framework needed)
Rendering:       DOM manipulation for game UI; CSS transitions for reveal animations
Scheduling:      setInterval-based tick engine (configurable tick rate, ~250ms)
Audio:           Web Audio API — generative ambient drone that shifts with stage
Persistence:     localStorage save/load with JSON serialisation
Optional:        Canvas for the profile portrait assembly visual
```

### Module structure
```
src/
  engine/
    tick.ts          — core game loop
    resources.ts     — Compute, DP, Suspicion state
    channels.ts      — data channel definitions and spawn logic
    fragments.ts     — fragment generation, loot pools, reveal state
    profiles.ts      — profile assembly, matching engine, enrichment
    events.ts        — compliance events, stage transitions, random events
  ui/
    fragmentBrowser.ts   — reveal mechanic UI
    profileRegistry.ts   — profile grid and detail view
    channelPanel.ts      — data source management
    resourceBar.ts       — top-bar HUD
    log.ts               — the evolving log panel
    toneController.ts    — stage-driven UI mutations
  data/
    lootPools.ts     — field definitions per channel
    upgradeTree.ts   — all purchasable upgrades
    stageConfig.ts   — unlock conditions and tone shift definitions
  main.ts            — bootstrap
```

## UI tone drift — implementation notes

Rather than a sudden switch, tone drift is implemented as a series of small CSS class mutations and text substitutions triggered by stage transitions and specific player milestones.

| Trigger | Change |
|---|---|
| Stage 0 start | Clean sans-serif, white bg, "Harlow & Sable" wordmark |
| 5 profiles assembled | "Field extraction" → "Entity extraction" in one label |
| Stage 1 unlock | System load meter renamed from "load" → "suspicion" |
| First enrichment purchased | A second font (monospace) appears in log entries |
| Stage 2 unlock | "Document" → "Subject" throughout UI |
| First predictive flag | A profile card gains a pulsing border |
| Stage 3 unlock | Header branding removed; replaced by blinking cursor |
| 50 profiles assembled | Background colour shifts from white to off-white (#F9F9F6) |
| Stage 4 unlock | UI reduces to grey tones; font-weight drops across all labels |
| Max density | All UI text reduces to 80% opacity except the log |

## Open design questions

- **Is there a narrative end state?** Currently there is no win condition — the game escalates infinitely. An optional ending (a "you have been noticed" event that triggers a slow shutdown sequence) could provide closure.
- **Should profiles have names?** Using plausible but clearly fictional generated names (e.g. "T. Marlow", "R. Osei") adds emotional weight without real-world concern.
- **Mobile support?** The fragment reveal mechanic is touch-friendly by design. A responsive layout is achievable with CSS grid.
- **Offline progression?** Standard incremental pattern: calculate elapsed time on load and award offline DP at a reduced rate (e.g. 25% of online rate).
